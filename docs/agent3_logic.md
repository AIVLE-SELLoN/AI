## 1. 파이프라인 · RAG 컬렉션 설계

```
[원인 분류 완료] (예: 색상 - 사진_색감_오차)
        ↓
[Agent3: 개선안 생성]
  ① 상세페이지 RAG 조회 (get, 필터: channel+product_group_i+aspect)
     → 이 채널의 이 aspect 관련 상세설명 내용 확인
  ② 과거 사례·반려 사례 RAG 조회 (query, 벡터 유사도)
     → 비슷한 케이스에 뭐가 효과적이었는지/반려됐는지
  ③ ①②를 종합해 개선안 텍스트 생성
     - 상세페이지에 근거 있으면: 인용해서 구체적으로
     - 근거 없으면(색상 등 텍스트로 검증 안 되는 경우): 확신도 낮추고 일반 가이드로
```

### RAG 컬렉션 설계

**컬렉션 1: 채널별 상세설명** 

- 상세설명을 통짜 텍스트로 넣지 않고 **aspect별 필드로 구조화**해서 저장
- Mock 스키마:  `input_detail_fields.csv`  [Mock 데이터 정의서 §5-1 참조](https://app.notion.com/p/Mock-39ee39c6149780d68c1bcbb5512787c4?pvs=21)
- ChromaDB 저장 시 채널·golden_group_id·aspect를 **메타데이터**로 붙여, 검색 시 `where` 필터로 정확히 좁힘
- 정확한 필드 조회는 `get`(필터만), 유사 사례 탐색은 `query`(필터+벡터유사도) 구분해서 사용
- **"기타" 필드는 벡터 검색(query) 대상에서 제외 — `get`으로만 조회 가능.** 미분류 텍스트라 벡터 유사도 품질이 낮음
- **이미지는 완전 미사용** (파일도, 캡션도 안 만듦)

**컬렉션 2: 과거 사례·반려 사례**

- 반려 시 사유를 이 컬렉션에 축적 → 다음 유사 케이스 개선안 생성 때 "이런 개선안은 반려됐었다" 참고

## 2. 정확도 방어책

| # | 방법 | 내용 |
| --- | --- | --- |
| 1 | **인용 검증(Grounding check)** | LLM이 "근거 있음"이라 답한 인용문이 실제 원문에 있는지 사후 검증. 없으면 확신도 자동 강등 |
| 2 | **확신도 점수화** | "높음/중간/낮음"을 느낌이 아니라 (CS근거+상세페이지근거+유사사례+시점일치) 가중합으로 계산 |
| 3 | **메타데이터+키워드 하이브리드 검색** | 순수 벡터 유사도만 쓰지 않고, aspect 필터 + 키워드 가중치 재순위화 병행 |
| 4 | **Self-consistency** | 같은 판정을 2~3회 반복해 답이 갈리면 확신도 하락 |
| 5 | **개선안 템플릿화** | 자유생성 대신 슬롯 채우기 방식으로 전환  |

## 4. 개발 상세 설계

### 4-1. 컬렉션 1 (채널별 상세설명) 스키마

- **문서 단위**: (golden_group_id × channel × aspect) 1건 = ChromaDB 문서 1건.
    
    Mock 소스는 `input_detail_fields.csv` 1행(총 504행 = 42상품 × 3채널 × 4aspect).
    
- **스키마** (input_detail_fields.csv, Mock 정의서 §5-1):

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| golden_group_id | string | 상품 단위 |
| channel | enum | COUPANG / NAVER / ZIGZAG |
| aspect | enum | 색상 / 사이즈 / 소재 / 기타 |
| detail_text | text | 실제 상세페이지 문구. 빈 값은 null 아닌 "정보 없음" |
- **임베딩 대상**: `detail_text` (query 벡터 유사도 검색용)
- **메타데이터**: golden_group_id · channel · aspect — 저장 시 부착, 검색 시 where 필터로 좁힘
- **조회 방식**: 정확 필드 조회 = get(필터: channel+golden_group_id+aspect) / 유사 사례 탐색 = query(필터+벡터유사도).
    
    "기타" aspect는 벡터 검색(query) 제외 — get으로만(미분류 텍스트라 유사도 품질 낮음)
    
- **정답 분리**: 그라운딩 정답은 이 컬렉션에 안 넣고 `golden_detail_fields.csv`로 분리 — Agent3 컨닝 방지.
- **적재**: Mock Producer 대상 아님 — `input_channel_products.csv`처럼 배치 초기적재.
    
    이미지 완전 미사용(파일·캡션 안 만듦)
    

### 4-2. 컬렉션 2 (과거 사례·반려 사례) 스키마

- **문서 단위**: 승인 또는 반려된 개선안 1건 = 문서 1건
- **검색용 텍스트**: "원인 라벨 + CS 요약 + 개선안 본문"을 합친 문자열
- **메타데이터**: channel / aspect / 원인라벨 / 결과(승인·반려) / **반려사유(반려 시 필수 — `reason_code+reason_text`, 출력 스키마 `hitl_feedback.rejection_reason`과 동일 구조)** / 일시
- **조회 방식**: aspect·원인라벨 필터 + 유사도 상위 3건.
    
    결과가 0건이면(초기엔 항상 0건) 유사 사례 블록을 숨기고 확신도를 한 단계 낮춤
    
- **성공 사례 데이터 출처**: "재촬영 → 2주 후 부정률 정상화" 같은 효과 정보는 승인 후 추적 기능이 스코프 밖이므로 **Mock에 미리 심어둔 사례로 제공** (실서비스 확장 시 추적 기능 필요, 문서화만)

### 4-3. Agent3 출력 구조

Agent3 출력 JSON의 **정본은 별도 문서 [「개선안 출력 스키마」](https://app.notion.com/p/39fe39c614978017b36befb4730e520c?pvs=21)**. 구조·필드·값 범위는 그 문서를 따른다.

- grounding 검사: 인용문이 실제 원문에 있는지 사후 검증 → 없으면 확신도 강등.
    - CS 인용=정본 `citations`(evidence.inquiry_ids 부분집합)
    - 상세페이지 인용=`proposal.current_text`로 구분.
- **grounding 소스는 도구별로 분리**: copy_draft → current_text를 상세페이지(detail_fields)와 대조 / image_guide → current_text가 CS 문의 기반 요약이므로 CS citations와 대조 (image_guide를 상세페이지로 검증하면 항상 실패).
- 출력이 형식에 안 맞으면 Evaluator 재시도 루프(최초 1 + 재시도 최대 2 = 총 3 시도)로 처리.

`proposal.proposed_text`는 자유생성이 아니라 슬롯 채우기(원인 라벨·근거 인용·권장 액션 슬롯)로 생성. 

슬롯 구성에 필요한 도구 선택은 [출력 스키마 §6 도구선택표](https://app.notion.com/p/39fe39c614978017b36befb4730e520c?pvs=21)를 따른다

### 4-4. 개선안 확신도 규칙

- **높음**: 상세페이지 근거 있음 + 유사 사례 있음
- **중간**: 둘 중 하나만 있음
- **낮음**: 둘 다 없음 → 일반 가이드 문구로 대체
- 승인·반려 데이터가 쌓여 "높음인데 반려되는" 케이스가 보이면 규칙 조정

산출은 **(CS근거+상세페이지근거+유사사례+시점일치) 가중합 점수**를 라벨로 매핑하는 방식으로 정의. 

데이터 부재로 초기 가중치는 규칙 기반 등가로 두고, 승인·반려 데이터가 쌓이면 재보정. (출력 스키마 `recommendation_confidence` 값 목록은 불변)

산출값은 [§5-1 탐지 캡핑](https://app.notion.com/p/39de39c61497809d8133dc4c2e36b319?pvs=21)을 거쳐 최종 확정

### 4-5. 예외 상황 처리 규칙

- 상세페이지가 등록 안 된 SKU → 근거-없음 경로 (확신도 낮음 + 일반 가이드)
- detail_fields 빈 값 → null이 아닌 "정보 없음"으로 저장, 생성 시 근거-없음 경로 강제
- 유사 사례 0건 → 4-2 규칙 적용 (블록 숨김 + 확신도 하향)

### 4-6. 원인 분류 라벨 목록·매핑

- 원인 분류 라벨 전체 목록과 라벨→aspect 매핑표는 [§5-2](https://app.notion.com/p/39de39c61497809d8133dc4c2e36b319?pvs=21)에 수록(Agent3 조회 필터의 근거).

## 5. 확신도 통합·라벨 매핑·정확도 실험

### 5-1. 확신도 이원화

- 시스템엔 확신도가 두 개 존재:
    - **이상탐지 확신도**([[7]단계](https://app.notion.com/p/3a3e39c614978081a40ddd7475e4d596?pvs=21) — 탐지·원인이 얼마나 확실한가)
    - **Agent3 확신도**(개선안이 얼마나 근거 있는가). *화면엔 개선안 확신도(Agent3)만 표시.*
- 캡핑은 [detection_confidence(탐지 확신도)](https://app.notion.com/p/3a3e39c614978081a40ddd7475e4d596?pvs=21) 필드로 입력. 단, Agent3는 `recommended_action="개선안 생성"`(=원인 명확) 게이트 때문에 실질적으로 **높음/중간**만 받는다.
- 따라서 실질 캡핑은 **탐지 중간 → 개선안 상한 중간(높음 표시 금지)**. 탐지 낮음·해당없음은 원인 미특정·전역/잠정전역이라 게이트에서 사전 차단(미도달).
    
    참조. 출력 스키마 [§3 capped_by_detection 필드](https://app.notion.com/p/39fe39c614978017b36befb4730e520c?pvs=21) + [§4 설계원칙](https://app.notion.com/p/39fe39c614978017b36befb4730e520c?pvs=21) 
    

**시점비교와의 관계 명확화**: [이상탐지 로직 [7]](https://app.notion.com/p/3a3e39c614978081a40ddd7475e4d596?pvs=21)의 `timestamp_matched`(상세페이지 수정시점 vs 이상 시작일 일치)는 **편중형+원인일관 케이스**에서 `detection_confidence`를 높음/중간으로 갈라놓는 실무 장치다.

이 결과는 위 캐핑 경로를 통해 Agent3에 간접 반영된다. Agent3 자신은 시점을 보지 않고 상세페이지 텍스트 내용(RAG)만으로 그라운딩한다

### 5-2. 원인 라벨 → aspect 매핑표

| aspect | 원인 라벨 후보 | detail_fields 대응 필드 |
| --- | --- | --- |
| 색상 | 사진_색감_오차 / 조명_보정_차이 / 실물_염색_편차 / 기타 | 색상 |
| 사이즈 | 표기_오타 / 실측_표기_편차 / 채널_사이즈_표준차이 / 기타 | 사이즈 |
| 소재 | 소재_정보_누락 / 이미지_질감표현_부족 / 실제_원단_문제 / 기타 | 소재 |
- 파손·오배송·기타 aspect는 [이상탐지 로직 [5] 스코프 필터](https://app.notion.com/p/3a3e39c614978081a40ddd7475e4d596?pvs=21)에서 개선안 생성 대상 제외(알림만 발행) → Agent3는 이 표의 3개 aspect만 처리하면 됨.
- detail_fields 스키마의 {색상, 사이즈, 소재, 기타} 필드 구성과 1:1 대응.
- 🚧 원인 라벨 안의 **"기타"**(사전정의 후보에 안 잡히는 경우)와 detail_fields의 "기타" 필드(상세페이지의 미분류 텍스트)는 다른 개념 — 변수명 충돌 방지 필요.

### 5-3. Mock 정확도 실험 설계

| 지표 | 계산 방법 | 데이터 소스 |
| --- | --- | --- |
| Retrieval hit rate | 컬렉션1 get 결과가 정답 필드와 일치 + 컬렉션2 query 상위 3건에 정답 문서 포함되는 케이스 수 / 전체 케이스 수 | Mock 정답 라벨 |
| Grounding precision | `proposal.current_text`(상세페이지 인용)가 detail_fields[aspect] 원문에 그대로 포함된 케이스 수 / `proposal.detailpage_grounded=true` 케이스 수 | Agent3 출력 로그 |
| 확신도 단계별 승인율 | 높음/중간/낮음 각각의 승인 비율 — 높음>중간>낮음 순서가 깨지면 [§5-1](https://app.notion.com/p/39de39c61497809d8133dc4c2e36b319?pvs=21)·[§4-4](https://app.notion.com/p/39de39c61497809d8133dc4c2e36b319?pvs=21) 규칙 재조정 신호 | HITL 로그 |
| 승인율 / 편집률 | 승인 건수 ÷ 전체, (승인 전 편집된 건수) ÷ 승인 건수 | HITL 로그 |
- **베이스라인 비교**: 동일 Mock 케이스셋을
    
    (A) 원인 라벨만으로 생성(RAG 미사용) vs (B) RAG 포함 생성
    
    두 조건으로 실행해 위 4개 지표를 나란히 비교. "RAG 도입으로 승인율 X%p·grounding precision Y%p 개선" 형태의 정량 주장 근거로 사용
    
- **HITL 로그 소스**: 별도 로그 스키마 없이 출력 스키마 필드로 정의 —
    - 케이스ID=`recommendation_id`
    - 확신도=`recommendation_confidence`
    - 승인/반려=`hitl_status`
    - 편집 여부=`hitl_feedback.edited_text≠null`
    - 반려 사유=`hitl_feedback.rejection_reason`
    
    위 실험은 이 필드들을 그대로 소비.
